# CDC 跨时钟域 — 理论笔记

## 1. 为什么要关注 CDC？

```
时钟域A（50MHz） → 发数据到 → 时钟域B（100MHz）
                          ↕
                  问题：A的信号变化时间，B可能采不到！
```

不同时钟域之间直接连信号 → **亚稳态（Metastability）** → 系统崩溃

---

## 2. 亚稳态（Metastability）

### 2.1 什么是亚稳态

D触发器结构上是个反馈环路，正常工作流程：

| 时间 | 行为 |
|------|------|
| 时钟沿来之前 | D端准备好 0 或 1 |
| 时钟沿来时 | 把 D 的值"拍"到 Q |
| 时钟沿之后 | Q 保持这个值直到下一个时钟沿 |

**亚稳态 = D 在时钟沿的"窗口期"内变化了 → 触发器无法判断是 0 还是 1 → 输出卡在中间电压**

```
clk  ▁▁▁▁▁▁▁▁▁▁▔▔▔▔▔▔▔▔▔▔  ← 上升沿
                              ↑
D    ▁▁▁▁▁▁▁▁▔▔▔▔▔▔▔▔▔▔▔  ← D也在同一时刻从0→1
                              ↑
Q = ??? （亚稳态）
```

### 2.2 亚稳态的波形

```
Q     ▁▁▁▁▁▁▁▁╲╱╲╱╲╱╲╱╲╱╲╱╲╱╲╱▔▔▔▔▔▔
               ↑  "震荡"阶段         ↑ 最终稳定到0或1
               不知道要多久          可能在这个时钟沿之后
```

### 2.3 类比理解

```
软件角度： 变量 = 不确定值   → 下次用它就崩了
硬件角度： 触发器的输出在震荡 → 给到下一级，整个链条都错了
```

### 2.4 概率问题：MTBF（Mean Time Between Failures）

亚稳态不是一定会发生，而是**概率**问题：

```
MTBF = e^(tMET / τ) / (fCLK × fDATA × C)
```

- `tMET`：留给输出稳定的时间（打了几拍）
- `τ`：工艺常数（约10~100ps）
- `fCLK`：时钟频率
- `fDATA`：数据变化频率

**关键结论：**

| 条件 | 亚稳态概率 |
|------|-----------|
| 直接连（0拍） | ~0.1%/次 → **每1秒可能崩1次** ❌ |
| 打1拍 | 降低到10⁻¹⁰/次 |
| **打2拍（两级同步器）** | **降低到10⁻²⁰/次 → 每千亿年崩1次** ✅ |
| 打3拍 | 没必要，除非极端高频 |

### 2.5 核心结论

> **亚稳态 = 信号在时钟采样窗口变化，触发器卡在中间状态。**
>
> **解决方法：打两拍（两级 D 触发器串联），等输出稳定了再用。**

## 3. 单比特同步器（慢→快）

### 3.1 电路结构

```
                    ┌────┐         ┌────┐
data_from_clkA ────→│FF1 │────────→│FF2 │────────→ data_to_clkB
                    └────┘         └────┘
                      ↑              ↑
                    clkB           clkB
```

就是两级 D 触发器串联，全部由**目标时钟域（clkB）**驱动。

### 3.2 为什么有效

- FF1 的输入可能亚稳态
- FF1 有**一个完整时钟周期（Tclk）**来稳定输出
- FF2 在下一个时钟沿采样时，FF1 已经稳定了
- FF2 输出的是干净的 0 或 1

### 3.3 Verilog 实现

```verilog
module sync_2ff #(
    parameter WIDTH = 1
)(
    input  clk,
    input  rst_n,
    input  [WIDTH-1:0] async_in,
    output [WIDTH-1:0] sync_out
);
    reg [WIDTH-1:0] sync_1, sync_2;

    always @(posedge clk or negedge rst_n) begin
        if(!rst_n) begin
            sync_1 <= {WIDTH{1'b0}};
            sync_2 <= {WIDTH{1'b0}};
        end
        else begin
            sync_1 <= async_in;
            sync_2 <= sync_1;
        end
    end

    assign sync_out = sync_2;
endmodule
```

### 3.4 缺点 / 限制

| 问题 | 说明 |
|------|------|
| **延迟 2 个时钟周期** | 数据从 clkA 到 clkB 要等 2 拍 |
| **脉冲可能丢失** | 如果 clkA 的脉冲宽度 < clkB 周期，可能采不到 |
| **只能传单比特** | 多位总线不能各自同步（违例问题，见后） |

### 3.5 核心问答

**问：打两拍后数据延迟了两个时钟周期，这个代价能接受吗？**

> 能接受。两个时钟周期非常短（比如50MHz时钟延迟40ns），在绝大部分设计中可以忽略。比起亚稳态导致系统崩溃，这个代价完全值得。这是**用时间换可靠性**。

**问：慢时钟域的脉冲传给快时钟域，打两拍后脉冲会变成什么样子？**

> 如果慢时钟域的信号保持时间足够（≥快时钟的2个周期），打两拍后脉冲宽度不变，只是延迟了2拍。但如果慢时钟信号变化太快（比如一个窄脉冲），快时钟可能采不到它 → **需要脉冲同步器**。

**问：为什么 CDC 不能对多 bit 总线各打各的拍？**

> 假设要传 8'b0000_1111，clkA 的 D 触发器在某个时刻输出从 0000_1111 变成 1111_0000，但每 bit 的延迟略微不同。clkB 可能在中间时刻采到 0000_0000 或 1111_1111 等错误值 → **数据一致性问题**。多位数据对齐必须用异步 FIFO 或握手协议。

## 4. 脉冲同步器（快→慢）

### 4.1 为什么需要它

快时钟域（clkA）的脉冲宽度可能只有 1 个 clkA 周期。慢时钟域（clkB）可能采不到它：

```
clkA  ▔▔▁▁▔▔▁▁▔▔▁▁▔▔▁▁▔▔
pulse ▁▁▁▁▁▔▁▁▁▁▁▁▁▁▁▁▁    ← 1个clkA周期

clkB  ▔▁▔▁▔▁▔▁▔▁▔▁▔▁▔▁    ← clkB比clkA慢得多
       ↑采样没碰到     ↑也没碰到 → 脉冲丢失！❌
```

### 4.2 解决方案：展宽 + 同步 + 边沿检测

原理是**先把脉冲转成电平（翻转触发器），传送电平，恢复出脉冲**：

```
                    ┌────┐        ┌────┐        ┌────┐
pulseA ─────┬──────→│FF1 │───────→│FF2 │───────→│FF3 │
            │       └────┘        └────┘        └────┘     
            │         ↑              ↑             ↑
            │       clkB           clkB          clkB
            │                                       │
            └───────────────────────────────────────┘
                 XOR ⨁  → pulseB
```

### 4.3 Verilog 实现

```verilog
module pulse_sync (
    input  clkB,
    input  rst_n,
    input  pulseA,
    output pulseB
);
    reg toggle, sync_1, sync_2, sync_3;

    // clkA 域：检测脉冲，翻转电平
    // 注意：这里假设 pulseA 来自 clkA 域，
    // 实际 clkA 域的 toggle 应该用 clkA 驱动
    // 但为了演示，简化为 posedge clkB 检测（实际不是这样）
    
    // 正确做法是分两个 always 块：
    // 1. clkA 域：检测到 pulseA 就翻转 toggle
    // 2. clkB 域：打两拍同步 toggle
    
    // 下面的完整实现需要跨域，详见实际代码

    // 同步到 clkB
    always @(posedge clkB or negedge rst_n) begin
        if(!rst_n) begin
            sync_1 <= 1'b0;
            sync_2 <= 1'b0;
            sync_3 <= 1'b0;
        end
        else begin
            sync_1 <= toggle;
            sync_2 <= sync_1;
            sync_3 <= sync_2;
        end
    end

    // 边沿检测恢复脉冲
    assign pulseB = sync_2 ^ sync_3;  // XOR → 检测翻转 → 输出一个周期的脉冲
endmodule
```

## 5. 异步 FIFO（多位数据传输）

### 5.1 为什么多位数据不能用打两拍

假设传 8 位数据 `0000_1111 → 1111_0000`：

```
clkA域:   0000_1111  →  1111_0000
clkB域(打两拍):
  bit0:  1 → 1 → 1  ← 立即可见
  bit1:  1 → 1 → 1
  bit2:  1 → 1 → 1
  bit3:  1 → 1 → 1
  bit4:  0 → 1 → 1  ← 走线稍长，晚1拍更新
  bit5:  0 → 0 → 1  ← 再晚1拍
  bit6:  0 → 0 → 0  ← 更晚
  bit7:  0 → 0 → 0

clkB 某个时刻看到：0000_1111 + 1111_0000 的混合体！ ❌
```

### 5.2 解决方案：格雷码

**格雷码的性质：每步只变1位**

```
二进制: 000 → 001 → 010 → 011 → 100 → 101 → 110 → 111
格雷码: 000 → 001 → 011 → 010 → 110 → 111 → 101 → 100
                         ↑ 每次只变1bit
```

**格雷码 + 打两拍**：即使同步时有一拍的误差，要么读到旧值要么读新值，不会读到"中间值" → **数据一致性保证** ✅

### 5.3 异步 FIFO 核心思想

生产者（写时钟域）往 FIFO 写数据，消费者（读时钟域）从 FIFO 读数据。问题在于**如何判断空/满**——这需要跨时钟域比较指针。

异步 FIFO 就是用**格雷码同步指针**，解决了跨时钟域的指针比较问题。

> 异步 FIFO 是面试手撕代码的**必考题**，我们后面 Week2 专门练。

## 6. 三大 CDC 方案总结

| 场景 | 方案 | 要点 |
|------|------|------|
| **慢→快，单比特** | 打两拍同步器 | 延迟2拍，不能输窄脉冲 |
| **快→慢，单比特** | 脉冲同步器 | 展宽→同步→沿检测 |
| **多位数据** | 握手 / 异步FIFO | 格雷码保证数据一致性 |

### 面试高频追问

1. **打两拍就够了吗？** 对大多数设计是的（MTBF足够大）。极端高频（>1GHz）可能需要3拍。
2. **打两拍的代价？** 延迟2个clk周期，绝大部分场景可接受。
3. **为什么不能用打两拍传多bit？** 每bit延迟不一致，读到"中间值"。
4. **异步FIFO深度怎么算？** 取决于写速率、读速率、突发长度。（Week2细讲）
